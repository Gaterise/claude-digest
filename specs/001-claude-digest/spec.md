# Feature Specification: Claude Digest

**Feature Branch**: `001-claude-digest`  
**Created**: 2026-04-14  
**Status**: Draft  
**Input**: User description: "claude codeのchange logが多く、人間にとって全てをキャッチアップするのは難しい問題を対策するアプリとしてClaude Digestというアプリを作成する。claudeのchange logの公式情報をなるべくリアルタイムに取得し、日本語にしてわかりやすく重要な点がまとめて記事のようにして閲覧ユーザーに見せたい。"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 最新ダイジェスト記事の閲覧 (Priority: P1)

閲覧ユーザーがアプリにアクセスすると、Claude の最新変更点が日本語で要約されたダイジェスト記事の一覧を確認できる。各記事はわかりやすく整理されており、重要なポイントが一目でわかる。

**Why this priority**: アプリの最重要価値はユーザーが Claude の変更点を素早く把握できることにある。この機能が動作していれば、それだけで MVP として成立する。

**Independent Test**: アプリを開いてダイジェスト記事一覧が表示される。最新の記事をクリックすると詳細が読める。

**Acceptance Scenarios**:

1. **Given** ユーザーがアプリのトップページにアクセスしたとき、**When** ページが読み込まれると、**Then** 最新の Claude 変更点ダイジェスト記事がリスト形式で表示される
2. **Given** ダイジェスト記事一覧が表示されているとき、**When** ユーザーが記事をクリックすると、**Then** 変更点の詳細が日本語でわかりやすく整理された記事ページが表示される
3. **Given** 記事詳細ページが表示されているとき、**When** ユーザーが内容を読むと、**Then** 重要なポイントが強調されており、公式変更ログの原文リンクも参照できる

---

### User Story 2 - 変更履歴の検索・フィルタリング (Priority: P2)

ユーザーが特定の期間や関心のある変更カテゴリでダイジェスト記事を絞り込んで確認できる。

**Why this priority**: 記事が蓄積されるにつれて、過去の特定の変更を探す手段が必要になる。P1 が完成した後に価値が生まれる機能。

**Independent Test**: 日付範囲や「バグ修正」「新機能」などのカテゴリでフィルタリングして、対象記事のみが表示されることを確認できる。

**Acceptance Scenarios**:

1. **Given** ダイジェスト記事一覧が表示されているとき、**When** ユーザーが日付範囲を指定すると、**Then** 指定期間内のダイジェスト記事のみが表示される
2. **Given** ダイジェスト記事一覧が表示されているとき、**When** ユーザーがカテゴリ（新機能・改善・バグ修正など）でフィルタリングすると、**Then** 該当カテゴリの変更を含む記事のみが表示される

---

### User Story 3 - 新しいダイジェストの通知受信 (Priority: P3)

ユーザーが新しいダイジェスト記事が公開されたときに通知を受け取れる設定ができる。

**Why this priority**: アプリを能動的に開かなくても変更点をキャッチアップできるようになり、ユーザー体験が向上する。コア機能が安定してから追加する。

**Independent Test**: 通知設定をオンにした状態で新しいダイジェストが生成されると、設定した方法（メール等）で通知が届くことを確認できる。

**Acceptance Scenarios**:

1. **Given** ユーザーが通知設定をオンにしているとき、**When** 新しいダイジェスト記事が生成されると、**Then** 設定した通知方法でアラートが届く
2. **Given** 通知が届いたとき、**When** ユーザーが通知をクリックすると、**Then** 該当のダイジェスト記事ページに直接遷移する

---

### Edge Cases

- 公式変更ログの取得元が一時的にアクセス不能な場合、最後に取得したデータを表示し「最終更新: XX分前」のように状態を示す
- 変更ログの内容が非常に少ない（または同一期間内に大量に更新された）場合でも、適切な粒度でダイジェストを生成する
- 日本語翻訳・要約の処理に時間がかかる場合、ユーザーに処理中であることを表示する
- 同一内容のダイジェストが重複生成されないよう重複チェックを行う

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: システムは Claude の公式変更ログを定期的に自動取得しなければならない（MUST）
- **FR-002**: システムは取得した変更ログを日本語に翻訳・要約し、わかりやすいダイジェスト記事として生成しなければならない（MUST）
- **FR-003**: システムは生成したダイジェスト記事を一覧形式でユーザーに表示しなければならない（MUST）
- **FR-004**: ユーザーはダイジェスト記事の詳細を閲覧できなければならない（MUST）
- **FR-005**: ダイジェスト記事には元の公式変更ログへの参照リンクを含まなければならない（MUST）
- **FR-006**: システムは変更ログの更新をなるべくリアルタイムに検知し、新しいダイジェストを生成しなければならない（MUST）
- **FR-007**: ユーザーはダイジェスト記事を日付範囲・カテゴリでフィルタリングできなければならない（SHOULD）
- **FR-008**: システムは変更点を「新機能」「改善」「バグ修正」「非推奨」などのカテゴリに自動分類しなければならない（SHOULD）
- **FR-009**: ユーザーは新しいダイジェスト公開時の通知設定を行えなければならない（COULD）
- **FR-010**: システムは同一内容のダイジェストが重複生成されないよう管理しなければならない（MUST）

### Key Entities

- **ChangeLog（変更ログ）**: 公式ソースから取得した生の変更ログデータ。取得日時、バージョン番号、原文コンテンツ、ソースURLを持つ
- **DigestArticle（ダイジェスト記事）**: 変更ログをもとに生成された日本語要約記事。タイトル、要約本文、重要ポイント、カテゴリ分類、生成日時、元ChangeLogへの参照を持つ
- **Category（カテゴリ）**: ダイジェスト記事を分類するラベル（新機能・改善・バグ修正・非推奨・その他）
- **NotificationSetting（通知設定）**: ユーザーごとの通知方法・頻度の設定

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 公式変更ログが公開されてから60分以内にダイジェスト記事が生成・公開される
- **SC-002**: ユーザーがアプリにアクセスしてから30秒以内に最新のダイジェスト一覧が表示される
- **SC-003**: ダイジェスト記事を読んだユーザーの90%以上が「変更点を理解できた」と評価する
- **SC-004**: 公式変更ログの重要な変更点の95%以上がダイジェスト記事に反映される
- **SC-005**: ユーザーが目的の記事を3クリック以内に発見できる

## Assumptions

- Claude の公式変更ログは公開されているウェブページまたはフィードとして取得可能であると仮定する
- 対象ユーザーは日本語話者であり、日本語のUIと記事コンテンツを前提とする
- 初期バージョン（v1）では登録なしでダイジェスト記事を閲覧できる読み取り専用の公開サービスを想定する
- 通知機能（P3）はv1スコープ外とし、後続バージョンで実装する
- AIによる要約・翻訳の品質はユーザーが変更点の概要を把握するのに十分なレベルを目標とし、完全な正確性よりも可読性を優先する
- モバイルブラウザからのアクセスも想定し、レスポンシブな表示を基本とする
